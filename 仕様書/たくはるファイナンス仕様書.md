## 4. データ構造（Firestore）

### 4.1 コレクション構成

```
users/{uid}
  ├─ householdId: "TAKUHARU"
  ├─ name: String（表示名）
  └─ fcmTokens: [String]（FCMトークン）

households/TAKUHARU（固定）
  ├─ members: [uid, uid]
  ├─ memberNames: {uid: "たく", uid: "はる"}
  ├─ memberIcons: {uid: "🐻", uid: "🐰"}
  ├─ paymentMethods: ["現金", "クレジットカード", ...]
  ├─ personalFoodBudgets: {uid: 8000, ...}（個人食費わく・月上限）
  ├─ replacements: [{"from": "...", "to": "..."}]（OCR辞書）
  ├─ customExpenseCats: ["カスタムカテゴリ"]
  ├─ customIncomeCats: [...]
  └─ monthlyBudget: int?（月予算・null=未設定）

  ├─ transactions/{id}    # 取引
  ├─ plan_items/{id}      # プランニング
  └─ subscriptions/{id}   # 固定費・サブスク
```

### 4.2 Transaction（取引）型定義

```dart
class Transaction {
  final String id;                  // ドキュメントID
  final DateTime date;              // 取引日
  final TransactionType type;       // expense / income / transfer
  final Category category;          // {major, sub}
  final String paymentMethod;       // "現金" 等
  final String description;         // 内容・摘要
  final int amount;                 // 金額（円・税込・常に正）
  final String? receiptUrl;         // Drive リンク
  final String? receiptId;          // 親レシート参照ID（OCR取込時）
  final String? memo;               // 備考・品目リスト
  final String? store;              // 店舗名
  final String? originalCurrency;  // "USD" 等（null=JPY）
  final double? originalAmount;    // 元通貨での金額
  final String? transferFromAccount;
  final String? transferToAccount;
  final bool isPending;             // 見込みフラグ（発生主義）
  final String? recordedBy;         // 記録者UID
  final String? paidBy;             // 支払者UID（割り勘用）
  final String? personalFor;        // 個人食費わく対象UID
  final int commentCount;           // チャット件数（読み取り専用）
  // Firestore: households/TAKUHARU/transactions/{id}
}

enum TransactionType { expense, income, transfer }
```

## 5. 認証・世帯共有の仕組み

### 5.1 認証フロー

```
AuthService.signInWithGoogle()
  → Android: GoogleSignIn.authenticate() → Firebase
  → Web: FirebaseAuth.signInWithPopup(GoogleAuthProvider)
  → HouseholdService.ensureHousehold(user)
      → households/TAKUHARU の members に uid を追加
      → 旧データがあれば _migrateData() で移行
```

### 5.2 世帯共有のポイント

- 世帯ID は **`TAKUHARU` 固定**（コード入力不要・ログインするだけで参加）
- `recordedBy: uid` で「誰が記録したか」を識別
- `paidBy: uid` で「誰が払ったか」を識別（割り勘用）
- `personalFor: uid` で「誰の個人食費わくか」を識別

### 5.3 Firestore セキュリティルール（概要）

- `users/{uid}`: 本人のみ読み書き
- `households/TAKUHARU` および配下: `members` に含まれるユーザーのみ

## 9. 収支入力の仕様

### 9.1 入力方式

1. **手入力** → `AddTransactionScreen` でフォーム入力
2. **レシート読取** → Gemini Vision API でOCR → 品目ごとか1件まとめかを選択

### 9.2 個人食費わく

- **対象カテゴリ（v0.2.91〜）**: 「食費」に加えて「日用品」も個人の食費わくに入れられる
  （レジ袋など食費と一緒に買う日用品が入ることがあるため）。判定は `categories.dart` の
  `isPersonalFoodCategory()`／`personalFoodCategories = {'食費','日用品'}` に集約し、登録画面・
  レシート品目記録・まとめ編集の各所で共通利用する。予算集計（ホームの個人食費わくカード）は
  カテゴリで絞らず `personalFor` だけで合算するので、日用品を入れても自動で反映される。
- 支出に `personalFor: uid` を付与 → その人の個人わくから引かれる
- 同時に世帯支出合計にも含まれる（共用財布から出る・個人の食費として記録）
- 月上限: `personalFoodBudgets[uid]`（既定 8,000円）
- **わくは常に「ログイン中の自分」のもの（v0.2.84〜）**: 相手のわくには入れられない。
  各画面で `personalFor` をセットするときは必ず自分の uid を使う（`_payer`/`_paidBy` ではない）。
  これにより「相手の食費わくを勝手に消費する」ことが起きない。
- **登録画面**: カテゴリで「食費」または「日用品」を選ぶと、カテゴリのすぐ下に目立つトグル
  「🍙 個人の食費わくから引く」が出る（v0.2.79〜／対象拡大 v0.2.91〜）。
  ラベルは自分の名前・自分の月わくを表示（v0.2.84〜）。相手のわくに入っている記録を
  編集すると「今は ◯◯ の個人わくに入っています（このトグルは自分のわく用です）」と注記し、
  自分の操作では相手のわくを外さない（`_resolvePersonalFor`）。
- **明細画面は表示だけ（v0.2.84〜）**: 食費の支出の明細を開くと、付いている場合は
  「◯◯ の個人の食費わくから」タグを表示するのみ。設定・変更・解除のボタンは廃止。
  付け外し・変更は**編集画面**で行う（明細からは操作しない）。
  ※旧「個人の食費わくに入れる／相手のわくに変更／はずす」ボタンは削除した。
- **レシート品目画面の一括設定（v0.2.84〜 / v0.2.85でチップ化）**: レシート読み取り後の
  「品目ごとに記録」（`ReceiptSplitScreen`）と「レシートを編集」（`ReceiptEditScreen`）で、
  食費の品目をまとめて個人わくに入れられる。**「品目（X件）」見出しの右隣に小さなチップ
  「🍙 ぜんぶ個人わく」**を置き、タップで全食費品目をON/OFF（個別カードのトグルと連動・
  全部ONならピンクに点灯）。わくは自分のもの。
  ※v0.2.84は全幅の大きいトグルだったが、見た目が重いため小さなチップに変更（v0.2.85）。

### 9.2.1 まとめレシートの詳細・編集（v0.2.79〜 / v0.2.90で「1レシート＝1画面」に統合）

- 同じ `receiptId` の品目が2件以上ある「まとめレシート」行（`ReceiptGroupTile`）は、
  **タップで `ReceiptDetailScreen`（レシート詳細画面）へ直行**する（v0.2.90〜。
  旧・その場で品目展開する挙動と ⋮ メニューは廃止）。
- **`ReceiptDetailScreen`（v0.2.90〜）**: レシート1枚を1画面にまとめる。
  上部にレシート概要（店名・日付・支払者・支払元・合計・レシート画像）→ 品目一覧（読み取り専用）→
  「編集」「削除」ボタン → その下に **1本のコメント欄**。
  品目ごとに詳細やコメントを持たず、**レシート単位に一本化**する。
- **「編集」= `ReceiptEditScreen`（v0.2.80〜）**: レシート登録時の `ReceiptSplitScreen` と
  同じ作りの**1画面**で、共通項目（日付・だれ・支払元・店名）も、品目ごと（品名・金額・カテゴリ・
  個人の食費わく）も**まとめて編集**できる。保存で「更新／追加（品目を足す）／削除（空にした品目）」を
  一括反映。全品目を空にすると確認の上レシートごと削除。編集後はレシート詳細に品目一覧が反映される。

#### カテゴリ内訳の明細もレシート単位（v0.2.95〜）

- ホーム「支出の内訳」と支出タブ「カテゴリ内訳」で、カテゴリ行をタップして開く明細も
  **1レシート＝1行**にまとめる（共通ウィジェット `ReceiptGroupedDetailList`／`receipt_group.dart`）。
  1枚のレシートに複数品目がぶら下がっていても、店名＋件数＋**そのカテゴリ分の合計**の1行になる。
- 行タップ → まとめレシートは `ReceiptDetailScreen`（品目は全カテゴリ分を表示）、
  単品は `TransactionChatScreen`。
- レシート判定は「同じ `receiptId` が月内に2件以上」。別カテゴリにまたがる品目も
  同じレシートとして数えるため、判定用に月の全取引（`allTxns`）を渡している。
- 支出タブのカテゴリ内訳は v0.2.95 で**開閉できるように**なった（それまでは棒グラフのみ）。

#### レシート単位のコメント（v0.2.90〜）

- 保存先は **`households/{hid}/receipts/{receiptId}/comments`**（`ReceiptCommentRepository`）。
  品目レコード（`transactions`）とは独立した場所に置くので、まとめ編集で品目を足し引きしても消えない。
- レシート本体 `receipts/{receiptId}` に `commentCount`（一覧の💬バッジ用）と `migratedFromItems`
  （旧・品目別コメントを統合済みか）を持たせる。
- **旧「品目ごと」に付いていたコメントの統合（移行）**: レシート詳細を初めて開いたとき、各品目の
  `transactions/{txId}/comments` を集めてレシートの1スレッドへ寄せ、元を削除、品目側の
  `commentCount` を0に落とす。`migratedFromItems=true` を立てて二度目以降はスキップ（冪等）。
- 一覧の💬バッジは「レシートの `commentCount`」と「品目側 `commentCount` の合計」の大きい方を表示
  （統合前は品目側、統合後はレシート側が生きる）。
- チャットUIは取引（単品）・レシート・プラン項目で共通の `CommentThread`（`lib/widgets/comment_thread.dart`、
  `CommentSource` 抽象で購読/投稿先を差し替え）に一本化。**単品取引（レシートでない支出/収入）は
  従来どおり `TransactionChatScreen` の品目単位チャット**（変更なし）。
- Firestore ルールに `households/{hid}/receipts/{receiptId}`＋`comments` を追加（メンバーのみ read/write）。

## 10. OCR レシート読み取り仕様

### 10.1 フロー

```
記録メニュー → "レシートを読取"
  → ReceiptCameraScreen（自前カメラ or ギャラリー）
  → 画像取得（品質55%・最大幅1024px）
  → Gemini gemini-2.5-flash へ base64 送信
  → Drive へ画像を並行アップロード（裏実行）
  → 記録方法を選択（まとめて1件 / 品目ごと）
  → AddTransactionScreen or ReceiptSplitScreen
  → 保存
```

### 10.2 Gemini が返す JSON

```json
{
  "store": "店名",
  "date": "YYYY-MM-DD",
  "amount": 1234,
  "category": "食費",
  "items": [
    { "name": "品名", "price": 500, "quantity": 2, "unitPrice": 250 }
  ]
}
```

### 10.3 まとめて1件 vs 品目ごと

| モード | 動作 |
|---|---|
| まとめて1件 | 合計金額で1件の支出。品目一覧を memo に記入 |
| 品目ごと | 各品目を独立した支出として記録 |

`ReceiptSplitScreen` の**店名は読み取り結果をテキスト編集できる**（v0.2.82〜。空も可、保存時に全品目へ反映）。

### 10.3.1 税込合計との差額調整（v0.2.86〜）

**背景の地雷**: レシートは常に「品目ごと」に分割記録するため、アプリの合計＝各品目priceの
単純合計になる。一方イオンネットスーパー等の**外税（税抜）レシートは品目が税抜き表記**で、
最下部にまとめて消費税が加算される。OCRは消費税の行を品目に入れない仕様のため、
**品目合計＝税抜小計となり、レシート印字の税込合計と「消費税ぶん」ズレていた**（v0.2.85まで）。
OCRは `amount`（税込最終支払額）を読み取っていたが、分割記録画面では完全に捨てていたのが原因。

**対策（v0.2.86）**: `ReceiptSplitScreen` 下部に**差額調整カード**を追加。
- **レシート合計（税込）**欄に OCR の `amount` をプリフィル（手修正可・未読取時は空欄で手入力可）。
- `差額 = レシート合計 − 品目合計` をライブ表示。
- 差額が 0 でなければ**スイッチ（既定ON）**で、差額を**1件の支出**として自動追加し記録合計を
  レシートとピッタリ合わせる。
  - 差額が正（税込>品目合計）→ 品目「**消費税・調整**」（カテゴリは下記のとおり v0.2.96 で変更）
  - 差額が負（値引き等で品目合計>税込）→ 品目「**値引き・調整**」（金額はマイナス）
  - 同じ `receiptId`・日付・支払元・支払者で追加するので、まとめ行に内包され内訳も残る。
- スイッチOFFなら従来どおり品目合計のまま記録（差額は記録しない）。

**編集画面でも同じ調整（v0.2.87〜）**: `ReceiptEditScreen`（まとめレシートの「レシートを編集」）にも
同じ差額調整カードを追加。**記録済みのレシートを後から税込合計に合わせ込める**。
- 「消費税・調整」「値引き・調整」の品目は**通常の品目リストには表示せず**、差額調整カードで一元管理する
  （`_isAdjustment` で判定して分離）。既存の調整行があれば、レシート合計欄に「全品目＋調整の合計」を
  プリフィルする。
- 保存時：差額があれば既存の調整行を**更新**（無ければ追加）。**差額が0になった／スイッチOFFにすると、
  既存の調整行は自動削除**（`keepIds` に入れない＝`deletes` 対象）。差額調整は通常品目が1件以上
  残るときだけ付く（全品目を空にした「丸ごと削除」には付けない）。
- **差額の品目カテゴリ＝そのレシートの「主たるカテゴリ」（v0.2.96〜）**: 以前は「その他」固定だったが、
  内訳に「その他」が並んで邪魔なので、**そのレシートの品目のうち金額合計が一番大きいカテゴリ**に寄せる
  （食費のレシートなら消費税ぶんも食費）。判定は `categories.dart` の `dominantCategory()`
  （`ReceiptSplitScreen`／`ReceiptEditScreen` の保存時に適用。調整行自身は判定から除外）。
- **過去分の直し（v0.2.96〜／v0.2.97で自動化）**: 既に「その他」で保存済みの調整行を主たるカテゴリへ
  一括で付け替える（`TxRepository.repairAdjustmentCategories`。金額・日付・品名は変更せず、
  何度実行しても安全）。**v0.2.97 からは `MainShell` の起動時に自動実行**（端末ごとに一度きり。
  SharedPreferences `takuharu.adjcat_repaired.v1` で管理し、失敗したら次の起動で再挑戦）。
  設定＞データ＞「消費税・差額のカテゴリを直す」の手動ボタンも残してある。
  ※Firestore への一括書き換えは開発PCからは実行できない（takuharu-finance の所有は
  takuharumika@gmail.com で、ツール環境には管理者資格情報が無い）ため、アプリ側で直す方式にした。

### 10.4 OCR辞書（表記ゆれ置換）

- `households/TAKUHARU/replacements` に `{from, to}` の辞書を保存
- OCR 読取後に自動で置換（例: 「セブン-イレブン」→「コンビニ」）

## 12. 通知（FCM）

### 12.1 登録・設定

```dart
// Androidチャンネル名: takuharu_default / 重要度: High
// 通知アイコン: ic_stat_takuharu（白シルエット貯金箱+ハート）
// 通知色: #FF6B8A（ピンク）
PushService.register() → 許可取得 → FCMトークン保存
```

### 12.2 タップ時の遷移

| payload | 遷移先 |
|---|---|
| `{type: "tx", txId: "...", receiptId: "..."}` | ReceiptDetailScreen（receiptId 優先・v0.2.94〜） |
| `{type: "tx", txId: "..."}` | TransactionChatScreen |
| `{type: "plan", planId: "..."}` | PlanDetailScreen |

`receiptId` が入っているのはレシートのまとめ通知（→12.4）。`listByReceiptId()` で品目を取り直して
レシート画面を開く。`txId` も併せて送っているので、旧バージョンのアプリでは従来どおり
先頭品目の TransactionChatScreen が開く（後方互換）。

### 12.3 通知の「部屋ごと上書き」＆「開いたら消す」（v0.2.93 で追加）

チャット的な挙動：**同じ取引/プランの通知は1枠にまとまり、その取引/プランを開くと通知が消える**。

- **タグで上書き（送信側 takuharu-notifier `notifier.py`）**: `send(..., tag=...)` で各通知に `tag` を付ける。
  同じ tag は端末側で1枠に上書き（Android=`AndroidNotification.tag`、前面表示=`data.tag`）。
  | 種類 | tag |
  |---|---|
  | 取引の記録/修正 | `tx:<txId>` |
  | レシートの記録/修正（まとめ・→12.4） | `receipt:<receiptId>` |
  | 取引へのコメント | `txcomment:<txId>` |
  | レシートへのコメント（v0.2.98〜） | `receiptcomment:<receiptId>` |
  | プランの追加/更新/完了 | `plan:<planId>` |
  | プランへのコメント | `plancomment:<planId>` |
  | 固定費の追加/変更 | `sub:<subId>` |
  （削除通知は対象の部屋が無いのでタグなし＝個別表示）
- **開いたら消す（アプリ側 `push_service.dart`）**: `TransactionChatScreen`/`PlanDetailScreen`/
  `ReceiptDetailScreen` の initState で `clearForTx(txId)` / `clearForPlan(planId)` /
  `clearForReceipt(receiptId)` を呼ぶ。`getActiveNotifications()` で配信済みを取り、
  該当 tag（`tx:`/`txcomment:`、`plan:`/`plancomment:`、`receipt:`）の通知を
  `cancel(id, tag)` で削除（Androidのみ）。
  - これで「開いたのに通知が残る→手で消す」がなくなる（LINE的）。たくはるカレンダーと同じ設計。

### 12.4 レシートは「1枚＝通知1件」にまとめる（v0.2.94 で追加）

レシートは品目ごとに取引が1件ずつ作られる（`TxRepository.addAll`）ため、素直に通知すると
**品目の数だけプッシュが飛ぶ**（実際に8品目のレシートで8連発していた）。送信側でまとめる。

- **まとめ待ち（`notifier.py`）**: `receiptId` を持つ取引は即送信せず
  `tx_pending[(receiptId, edited)]` に貯め、`threading.Timer` で `RECEIPT_BATCH_SEC`（5秒）待つ。
  品目が届くたびにタイマーを張り直し、**最後の1件から5秒静かになったら1件だけ送る**。
  （Firestoreのbatch commitは通常1スナップショットで届くが、分割して届いても取りこぼさない）
- **文面**: 2件以上 → `「◯◯ がレシートを記録したよ ♡」`／本文 `店名　N品目　-¥合計`。
  1件だけのとき（手入力＋画像の `detail_` レシートなど）は従来どおりの単品文面。
- **編集も同じ**: まとめ編集で複数品目が一度に変わっても `「レシートを修正したよ」` 1件。
  `(receiptId, edited)` をキーにするので「記録」と「修正」は混ざらない。
- `receiptId` の無い単発の取引・コメント・プラン・固定費の通知は**従来どおり即送信**（遅延なし）。

### 12.5 明細を直したらチャットに「変更履歴」を残す（v0.2.98 で追加）

たくはるカレンダーの「〇〇が予定を変更しました（時刻・場所）」と同じ仕組みを明細に入れた。
**編集通知はこの変更履歴コメント一本に統一**し、「何が変わったか」まで通知本文に載せる。

- **書き出し（アプリ側）**: 編集の入口は2つだけ。
  | 編集画面 | 書き込み先 | 対象項目 |
  |---|---|---|
  | `AddTransactionScreen`（単品編集） | `transactions/{txId}/comments` | 金額・品名・カテゴリ・日付・支払い・お店・払った人・個人わく |
  | `ReceiptEditScreen`（レシートまとめ編集） | `receipts/{rid}/comments` | 共通項目は1回だけ＋品目ごとの変更／追加／削除 |
  差分は `lib/data/tx_diff.dart` の `txDiff()`。`TxField` で「レシート共通項目
  （`kTxCommonFields`）」と「品目ごとの項目（`kTxItemFields`）」を出し分ける。
  変更ゼロなら**何も書かない**（＝通知も飛ばない）。
- **データ**: コメントdocに `kind: 'log'` を持たせる（既存＝会話は `kind` 無し＝`comment` 扱い）。
  **`commentCount` は増やさない**（💬バッジは会話用のまま）。
- **表示（`comment_thread.dart`）**: `kind='log'` は吹き出しではなく**中央のグレー帯**（カレンダーと同じ）。
- **通知（`notifier.py`）**: `_handle_cm` が `kind='log'` を検知 →
  `「〇〇が記録を直したよ ✏️」`／本文＝変更点（例 `個人わく なし → たく`）。
  tag は `tx:<txId>` / `receipt:<rid>` ＝記録通知と同じ枠に上書き。ON/OFFは記録系(`tx`)のトグル。
- **⚠ 取引の MODIFIED 監視は廃止した**。以前は Firestore の更新を見て「記録を修正したよ」を送っていたが、
  ①`personalFor`（個人わく）を見ていないので**個人わくを変えても通知が飛ばなかった**
  ②`receiptUrl` の後付けやカテゴリ自動修復など**内部的な書き換えでも通知が飛びうる**、の2つの穴があった。
  変更履歴コメント基点に変えて両方解消。
- **同時に直したバグ**: `notifier.py` が `category.major` を読んでいたが、実データは
  `categoryMajor`（フラット）なので**常に空文字**だった → `categoryMajor` に修正。
  レシートのチャットへのコメント通知が `txId=receiptId` で飛んでいて**タップしても開けなかった**
  → `receiptId` を送るよう修正（tag も `receiptcomment:<rid>`）。
